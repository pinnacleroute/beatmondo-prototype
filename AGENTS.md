# Prototype Instructions

Run the local server yourself and open the preview in the in-app browser. Do not give the user server-start instructions when you can run it.

Before making substantial visual changes, use the Product Design plugin's `get-context` skill when the visual source is unclear or no longer matches the current goal. When the user gives durable prototype-specific design feedback, preferences, or decisions, record them in `AGENTS.md`.

When implementing from a selected generated mock, treat that image as the source of truth for layout, component anatomy, density, spacing, color, typography, visible content, and hierarchy.

Durable Beatmondo direction: treat the product as a premium, custom-built music discovery and sync licensing platform, not a basic website or stock-music library. Keep black and gold as brand accents inside a broader editorial system with warm ivory, charcoal, taupe, champagne, soft grey, image-led sections, and restrained cinematic overlays. Surfaces should support professional buyers, artists, partners, and investors with rights-aware metadata, project workflows, preview-only listening, protected master/WAV delivery, audit-ready admin states, and respectful Gary Burke legacy/archive content.

Durable terminology direction: use "Explore Music" for buyer-facing discovery and reserve "Catalog" for admin/internal operations. Use "Request License" for track/project licensing CTAs and "Request Access" only for account/workspace access. Keep availability, workflow status, delivery state, and admin review state distinct; use "Quote Required" for track availability and "Needs Quote" for internal pipeline status. Prefer "protected master audio" in explanatory copy and "WAV master" only when naming the delivered file. Use "Stories & Media" as the content-area label instead of Blog/Podcast naming.
